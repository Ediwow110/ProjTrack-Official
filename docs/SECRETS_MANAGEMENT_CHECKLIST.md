# Secrets Management Checklist

Branch: `2nd-main`  
Last updated: 2026-05-14

## Verdict

Not fully accepted yet.

This checklist defines the minimum controls required for application secrets, deployment credentials, CI credentials, token-signing material, provider credentials, storage credentials, and webhook verification material.

## Required controls

### Storage and access

- [ ] Production secrets are stored only in the deployment provider, protected environment, or approved secret manager.
- [ ] No production secrets are committed in source, docs, screenshots, fixtures, logs, examples, or local environment files.
- [ ] Local environment files are ignored by git.
- [ ] CI secrets are scoped only to workflows that require them.
- [ ] Deployment workflows, if added, use protected environments and approval rules.
- [ ] Stale CI and deployment secrets are removed.

### Strength and separation

- [ ] Token-signing values are long, random, and separated by purpose.
- [ ] Account-action encryption material has the required length and encoding.
- [ ] Database credentials are environment-specific and not shared with developer machines.
- [ ] Object-storage credentials are least-privilege and bucket-scoped.
- [ ] Mail-provider credentials are scoped where the provider supports scoping.
- [ ] Smoke-test credentials are test-only and not reused human admin credentials.

### Runtime checks

- [x] Runtime safety checks flag weak or reused token-signing values.
- [x] Runtime safety checks flag missing account-action encryption material.
- [x] Runtime safety checks flag local production database, local production storage, unsafe rate-limit storage, and unsafe upload scanning configuration.
- [x] Runtime safety checks are covered by security tests.
- [ ] Live staging or production-safe runtime check output is recorded without secret values.

### Logging and redaction

- [x] Health/config redaction tests exist.
- [ ] Application logs redact authorization headers, cookies, session tokens, account-action tokens, provider credentials, database URLs, and storage credentials.
- [ ] Failed-auth logs contain useful forensic metadata without secret values.
- [ ] Provider errors are sanitized before being returned through public or user-facing endpoints.

### Rotation

- [ ] Token-signing rotation procedure exists.
- [ ] Refresh-session invalidation behavior is documented.
- [ ] Account-action encryption rotation procedure exists.
- [ ] Database credential rotation procedure exists.
- [ ] Object-storage credential rotation procedure exists.
- [ ] Mail-provider credential rotation procedure exists.
- [ ] Webhook verification material rotation procedure exists.

## Rotation record template

Use this format for each rotation event:

```text
Secret category:
Storage location:
Generated by:
Applied by:
Validation command:
Rollback plan:
Old credential revoked at:
Owner:
Date:
```

## Pre-production checklist

- [ ] Secret scan passes.
- [ ] Backend security tests pass.
- [ ] Runtime configuration check passes without exposing secret values.
- [ ] No committed local environment file contains real credentials.
- [ ] CI secrets are reviewed.
- [ ] Deployment-provider secrets are reviewed.
- [ ] Smoke credentials are test-only.

## Fail conditions

`SEC-GATE` fails if:

- A real secret is committed.
- Token-signing material is weak, reused across purposes, or missing.
- Production uses local file storage or memory-only rate limiting.
- Health/config endpoints reveal secret values.
- Required rotation procedures do not exist.
